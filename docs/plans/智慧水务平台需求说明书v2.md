## 二、项目概述

### 2.1 项目背景

随着城市化进程加快和水资源管理精细化要求不断提升，传统水务管理模式已难以满足集团化、数字化运营需求。环保集团下属多家水务子公司在设备管理、生产运营、安全监管等方面存在数据孤岛、流程不统一、管理效率低等问题。

本项目旨在构建一套统一的智慧水务平台，以逻辑多租户架构支撑集团总部与各子公司的协同管理，实现数据互通、流程规范、决策智能。

### 2.2 建设目标

1. **统一管理平台**：构建集团与子公司统一使用的 Web 管理平台，支持集团总览视角与子公司独立运营视角
2. **多租户隔离**：基于逻辑多租户架构，各子公司数据相互隔离，集团可跨租户汇聚分析
3. **业务全覆盖**：涵盖设备、监测、生产、安全、物资、经营、管网、AI 助手等核心业务模块
4. **决策支持**：通过数据看板、趋势分析、AI 智能助手提升管理决策效率
5. **标准化流程**：工单、两票、值班、巡检等核心业务流程实现线上化、标准化

### 2.3 技术栈

## 三、多租户架构需求

### 3.1 租户类型定义

| 租户类型 | 租户ID | 说明 |
|---------|--------|------|
| 集团总览 | `group` | 汇聚所有子公司数据，展示汇总指标和跨租户对比，只读聚合视角 |
| 广州水务 | `gz` | 广州子公司独立运营数据 |
| 深圳水务 | `sz` | 深圳子公司独立运营数据 |
| 东莞水务 | `dg` | 东莞子公司独立运营数据 |

### 3.2 租户切换机制

- 顶部 Header 提供租户选择器下拉组件，显示当前租户名称和 Logo
- 切换租户后，页面所有业务数据联动刷新，URL 路径保持不变
- 当前租户状态持久化存储于 Pinia Store（`useTenantStore`），页面刷新后恢复
- 集团视角下，新增/编辑/删除等写操作按钮隐藏或禁用，仅支持查看
- 权限系统根据 `currentTenantId` 和用户角色联合判断操作权限

### 3.3 Pinia Store 设计

```ts
// src/store/modules/tenant.ts
interface TenantState {
  currentTenantId: string   // 'group' | 'gz' | 'sz' | 'dg'
  tenantList: Tenant[]
  isGroupView: boolean       // 是否集团视角
}

interface Tenant {
  id: string
  name: string
  logo?: string
  type: 'group' | 'subsidiary'
}
```

### 3.4 Mock 数据规范

- Mock 数据文件按 `tenantId` 组织，路径格式：`src/mock/{module}/{tenantId}.ts`
- 集团视角（`group`）返回聚合数据，数值为各子公司求和或平均
- 接口请求统一携带 `tenantId` 参数（Header 或 Query Param）
- 跨租户对比接口同时返回所有子公司的对比数组

## 四、功能模块详细需求

### 4.1 驾驶舱模块

#### 4.1.1 集团看板

**功能描述**：集团视角下的核心运营指标汇总展示页面。

**页面布局**：
- 顶部：4个核心 KPI 卡片（供水量、水质达标率、设备在线率、告警数量）
- 中部左：各子公司供水量折线趋势图（近30天）
- 中部右：水质达标率柱状图（按子公司对比）
- 底部左：设备故障分布饼图
- 底部右：实时告警列表（最新10条）

**数据要求**：
- KPI 卡片显示当日数据，附环比变化箭头（上升/下降/持平）
- 图表支持按时间范围筛选（今日/本周/本月/自定义）
- 集团视角展示所有子公司汇总，支持下钻到单一子公司
- 数据自动刷新间隔：5分钟

**交互需求**：
- KPI 卡片点击可跳转至对应详情模块
- 告警列表点击跳转至报警管理页
- 图表支持 tooltip、缩放、导出图片

#### 4.1.2 租户对比

**功能描述**：多租户（子公司）横向对比分析页面。

**页面布局**：
- 顶部：指标选择器（可切换对比维度：供水量/水质/设备/告警/能耗）
- 中部：雷达图（展示各子公司在5个维度的综合评分）
- 底部：对比表格（各子公司在选定指标下的详细数据，支持排序）

**数据要求**：
- 支持选择对比周期（本月/上月/本季度/自定义）
- 雷达图各维度数值归一化处理（0-100分）
- 表格支持导出 Excel

- 整改任务指派和进度跟踪
- 复查确认闭环
- 隐患统计分析（按类型/部门/时段）

#### 4.5.2 安全巡检

- 巡检路线和点位配置
- 巡检计划下发（按日/周/月）
- 巡检执行记录（扫码/NFC 打卡 + 现场拍照）
- 漏检/未按时完成自动提醒

#### 4.5.3 应急预案

- 预案文档上传和版本管理
- 预案分类（设备故障/水质异常/自然灾害等）
- 演练计划和演练记录
- 预案关联告警事件

#### 4.5.4 事故记录

- 事故基本信息登记（时间、地点、类型、经过）
- 调查报告附件上传
- 处理结果和整改措施记录
- 事故统计分析

## 五、非功能性需求

### 5.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面首屏加载 | ≤ 3s（生产环境） |
| 列表查询响应 | ≤ 1s |
| 图表渲染 | ≤ 2s |
| 并发用户数 | ≥ 100 |

### 5.2 兼容性要求

- 浏览器：Chrome 90+、Edge 90+、Firefox 88+
- 屏幕分辨率：最低 1366×768，推荐 1920×1080
- 不要求移动端适配（本期）

### 5.3 安全要求

- 登录需要账号密码认证，支持记住登录状态
- 基于角色的菜单和按钮级权限控制
- 操作日志记录（关键操作可追溯）
- 接口请求携带 Token 认证

### 5.4 可用性要求

- 系统可用率 ≥ 99.5%（生产环境）
- 关键操作提供二次确认
- 表单数据有本地草稿缓存（防止误刷新丢失）
