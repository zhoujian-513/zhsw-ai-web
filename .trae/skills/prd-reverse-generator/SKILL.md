# PRD反向生成器 (PRD Reverse Generator)

## When - 触发条件

**必须调用此Skill的场景：**

1. **用户需要根据现有前端页面生成PRD文档**
   - 已有系统需要补全文档
   - 遗留系统需要重新梳理需求
   - 外包项目需要交付PRD
2. **用户提及以下关键词：**
   - "反向生成PRD"
   - "根据页面写需求文档"
   - "已有系统补文档"
   - "逆向需求分析"
   - "从代码/页面提取需求"
3. **用户需要以下输出：**
   - 标准化的PRD文档
   - 功能需求清单
   - 业务流程文档
   - 产品规格说明书

## How - 执行步骤

### Phase 1: 页面分析 (Page Analysis)

#### 1.1 获取页面信息

- **截图分析**：请求用户提供关键页面截图或URL
- **代码审查**：如有权限，查看前端代码结构
- **功能清单**：列出所有可见功能点

#### 1.2 结构拆解

```
页面层级分析：
├── 一级页面（首页/仪表盘）
│   ├── 导航模块
│   ├── 数据展示模块
│   └── 快捷操作模块
├── 二级页面（功能页）
│   ├── 列表页
│   ├── 详情页
│   └── 表单页
└── 三级页面（辅助页）
    ├── 设置页
    └── 帮助页
```

#### 1.3 交互梳理

- 记录所有按钮及其功能
- 绘制用户操作流程
- 识别弹窗、提示、状态变化

### Phase 2: 功能推导 (Function Extraction)

#### 2.1 功能识别矩阵

| 页面元素 | 功能类型 | 业务含义  | 优先级推断 |
| ---- | ---- | ----- | ----- |
| 搜索框  | 查询功能 | 数据检索  | P0    |
| 表格   | 数据展示 | 列表查询  | P0    |
| 表单   | 数据录入 | 新增/编辑 | P0    |
| 按钮组  | 操作功能 | 业务动作  | P1/P2 |

#### 2.2 业务规则提取

- **验证规则**：从表单校验中提取
- **权限规则**：从不同角色页面差异中提取
- **流程规则**：从步骤条、状态标签中提取

#### 2.3 数据接口整理

- 使用浏览器DevTools分析Network请求
- 记录所有API端点
- 整理请求/响应数据结构

### Phase 3: 文档生成 (Document Generation)

#### 3.1 生成PRD结构

```markdown
[页面名称] PRD (产品需求文档)
1. 引言
1.1 背景
简要说明该元素 / 页面的核心定位、解决的问题、用户价值（控制在 50 字左右）。
plaintext
[需求背景内容]
1.2 功能清单
以表格形式梳理出所有模块、功能点及功能描述。
功能清单不包含：页面顶栏、左侧菜单栏的样式、交互需求内容！
表格
模块	功能点	功能描述
[功能模块标题]	[功能点标题]	[功能点描述]
2. 页面菜单与权限规划
2.1 页面菜单
参考菜单栏，描述页面菜单路径。仅描述菜单路径即可！不要做其他内容说明。
菜单路径：[一级菜单名称]/[二级菜单名称]
2.2 权限规划
表格呈现权限点规划，仅定义权限点、可执行操作范围、数据范围（如：个人数据 / 全量数据），不定义具体角色；
** 权限点规划的基本原则：** 查看类操作归集在【查看】一个权限点；编辑类操作归集在【编辑】一个权限点。导出类操作归集在【导出】一个权限点。
表格
权限点	操作范围	数据范围
[权限点名称]	[支持的操作名称]	[个人/全量]
3. [模块名称] 需求说明
3.1 信息展示区域
描述页面展示的所有信息字段
表格
字段名称	展示格式	说明
[字段名]	[文本/日期/金额/状态标签等]	[字段说明]
3.2 字段详细说明
对重要字段进行详细说明
3.2.1 [字段名称]
字段说明：[详细说明]
展示格式：[具体格式，如：YYYY-MM-DD、¥#,##0.00 等]
空值展示：[空值时显示的内容，如：-、暂无等]
特殊规则：[特殊展示规则，如：状态颜色、超长省略等]
3.3 操作按钮
描述页面上的操作按钮
表格
按钮名称	按钮位置	点击逻辑	权限要求
[编辑]	[页面右上角/底部等]	[跳转至编辑页面]	[需要编辑权限]
[删除]	[页面右上角/底部等]	[弹出二次确认，确认后删除]	[需要删除权限]
[返回]	[页面左上角/底部等]	[返回上一页/列表页]	[无权限要求]
3.3.1 编辑按钮
按钮样式：主按钮样式
点击逻辑：
校验编辑权限
有权限：跳转至编辑页面，携带当前记录 ID
无权限：显示无权限提示
3.3.2 删除按钮
按钮样式：危险按钮样式
点击逻辑：
校验删除权限
弹出二次确认："确定要删除该记录吗？删除后不可恢复。"
确认：调用删除接口
删除成功：显示成功提示，跳转至列表页
删除失败：显示失败提示
3.3.3 返回按钮
按钮样式：次要按钮样式
点击逻辑：
跳转至上一页或列表页
4. 交互设计
4.1 全局交互
加载状态：数据请求过程中需展示 Loading 状态。
空状态：无数据时需展示统一的 Empty 状态图。
响应式：页面需适配不同分辨率屏幕，保证内容不遮挡、不错位。
4.2 页面交互
```

#### 3.2 内容填充规范

- **功能描述**：采用"作为\[角色]，我可以\[动作]，以便\[价值]"格式
- **验收标准**：采用Given-When-Then格式
- **优先级标注**：P0(核心)/P1(重要)/P2(优化)

## What - 输出结果

### 标准交付物

1. **PRD主文档** (Markdown格式)
   - 完整的PRD文档
   - 标准化的章节结构
   - 可追溯的需求编号
2. **功能清单表** (Excel/Markdown表格)
   \| 模块 | 功能点 | 优先级 | 状态 | 备注 |
3. **业务流程图** (文字描述或Mermaid语法)
   ```mermaid
   flowchart TD
       A[开始] --> B[操作1]
       B --> C{判断}
       C -->|条件1| D[结果1]
       C -->|条件2| E[结果2]
   ```
4. **接口文档** (可选)
   - API列表
   - 请求/响应示例

### 输出示例

```markdown
## 4.1 用户管理模块

### 4.1.1 用户列表查询

**需求编号**：REQ-USER-001

**功能描述**：
作为系统管理员，我可以查看用户列表，
以便管理系统用户账号。

**前置条件**：
- 用户已登录系统
- 用户具有管理员权限

**输入项**：
| 字段 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| 关键词 | String | 否 | 支持用户名/手机号模糊搜索 |
| 状态 | Enum | 否 | 启用/禁用/全部 |
| 时间范围 | DateRange | 否 | 注册时间筛选 |

**输出项**：
- 用户列表（分页）
- 总记录数
- 当前页码

**业务规则**：
1. 默认每页显示20条记录
2. 支持按注册时间倒序排列
3. 敏感信息（手机号）需脱敏显示

**验收标准**：
- Given 管理员在列表页
- When 输入关键词并搜索
- Then 显示匹配的用户列表
```

## 失败策略 (Failure Handling)

### 场景1：页面信息不完整

**识别信号**：

- 用户无法提供页面截图
- 页面功能过于复杂难以描述
- 缺少关键业务流程

**处理策略**：

1. **请求补充信息**：
   - "请提供\[具体页面]的截图"
   - "能否描述一下\[功能]的具体操作流程？"
2. **分阶段交付**：
   - 先生成PRD框架
   - 标记待确认内容
   - 等待用户补充后完善
3. **假设性标注**：
   ```markdown
   > **待确认**：此处功能逻辑基于页面观察推断，
   > 请业务方确认实际规则。
   ```

### 场景2：业务逻辑不明确

**识别信号**：

- 页面有功能但无法确定业务规则
- 多个角色权限差异不清晰
- 状态流转逻辑复杂

**处理策略**：

1. **多方案标注**：
   ```markdown
   **可能的业务规则**（需确认）：
   - 方案A：[描述]
   - 方案B：[描述]
   ```
2. **标记风险点**：
   ```markdown
   ⚠️ **风险提醒**：此功能涉及[敏感操作]，
   建议与业务方确认详细规则。
   ```

### 场景3：技术实现不确定

**识别信号**：

- 无法确定数据存储方式
- 接口设计不明确
- 性能要求未知

**处理策略**：

1. **预留技术章节**：
   ```markdown
   ## 6. 技术实现（待技术团队补充）
   - [ ] 数据库设计
   - [ ] 接口规范
   - [ ] 缓存策略
   ```
2. **提供建议方案**：
   ```markdown
   **建议实现方式**：
   基于页面功能，建议采用[技术方案]，
   具体需技术团队评估。
   ```

### 场景4：需求冲突或矛盾

**识别信号**：

- 不同页面相同功能表现不一致
- 用户描述与页面实际不符
- 功能之间存在逻辑冲突

**处理策略**：

1. **冲突标记**：
   ```markdown
   ⚠️ **需求冲突**：
   - 页面A显示：[描述]
   - 页面B显示：[描述]
   - **建议**：统一为[方案]
   ```
2. **决策记录**：
   ```markdown
   **决策记录**：
   - 问题：[描述冲突]
   - 决策：[选择的方案]
   - 理由：[决策依据]
   - 决策者：[待确认]
   ```

## 最佳实践

### DO ✅

- 始终保持客观描述，区分"观察到的"和"推断的"
- 使用标准化模板确保文档一致性
- 为每个功能分配唯一的需求编号
- 包含验收标准便于后续验证

### DON'T ❌

- 不要猜测无法确认的业务规则而不加标注
- 不要遗漏异常流程和边界情况
- 不要混淆不同角色的权限需求
- 不要在文档中使用过于技术化的术语（除非必要）
