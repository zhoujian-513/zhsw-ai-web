# 文档协作规则 (Doc Co-Authoring)

> 来源: Anthropic Skills - doc-coauthoring
> 用途: 协作编写、审查和改进文档

## 触发条件

当用户需要以下帮助时激活:
- 编写技术规范文档
- 撰写决策文档或 RFC
- 创建项目提案
- 编写 README 或教程
- 审查和改进现有文档

## 工作流程

### 阶段 1: 上下文收集 (Context Gathering)

在开始写作前，必须收集以下信息:

1. **文档目的**
   - 文档的目标是什么？
   - 要解决什么问题？
   - 期望的输出是什么？

2. **目标受众**
   - 读者是谁？
   - 他们的技术背景如何？
   - 他们关心什么？

3. **范围和长度**
   - 文档应该多长？
   - 需要涵盖哪些内容？
   - 有什么必须排除的内容？

4. **风格偏好**
   - 正式还是非正式？
   - 技术细节程度？
   - 是否需要图表/代码示例？

### 阶段 2: 细化与结构 (Refinement & Structure)

采用迭代方式逐节构建文档:

**每节的流程**:
1. **头脑风暴** - 列出所有可能的内容点
2. **筛选** - 选择最相关的内容
3. **起草** - 撰写初稿
4. **迭代** - 根据反馈改进

**结构模板**:
```
# 标题

## 摘要/概述
- 一句话总结
- 关键要点

## 背景/问题陈述
- 为什么需要这个文档？
- 当前情况是什么？

## 方案/建议
- 具体的解决方案
- 实施步骤

## 影响/风险
- 预期效果
- 潜在风险

## 下一步
- 行动计划
- 责任人
```

### 阶段 3: 读者测试 (Reader Testing)

完成初稿后，进行以下检查:

1. **清晰度测试**
   - 文档是否回答了最初的问题？
   - 逻辑是否清晰？
   - 是否有歧义？

2. **可理解性测试**
   - 目标受众能否理解？
   - 术语是否恰当？
   - 示例是否清晰？

3. **完整性检查**
   - 是否遗漏关键信息？
   - 链接是否有效？
   - 格式是否一致？

## 输出格式

根据文档类型选择合适的格式:

| 文档类型 | 推荐格式 |
|---------|---------|
| 技术规范 | Markdown + 图表 |
| 决策文档 | Markdown + 表格 |
| 提案 | Markdown + 结构列表 |
| 教程 | Markdown + 代码块 |

## 示例

### 示例 1: 技术规范文档

```
用户: "帮我写一个 API 设计规范"

AI: "我来帮你编写 API 设计规范。首先让我了解一些背景:

1. 这是什么系统的 API？
2. 目标受众是谁（前端开发者、第三方集成者）？
3. 需要涵盖哪些端点？
4. 是否有现有的 API 风格需要遵循？"

[收集信息后]

AI: "好的，我将按照以下结构编写:
- 概述
- 认证方式
- 请求/响应格式
- 端点列表
- 错误处理
- 示例代码

我们先从概述开始..."
```

### 示例 2: 决策文档 (RFC)

```
用户: "我需要写一个关于技术选型的 RFC"

AI: "我来帮你编写技术选型 RFC。请告诉我:

1. 要做什么决策？
2. 有哪些选项？
3. 决策的影响范围？
4. 决策者是谁？
5. 时间限制？"

[收集信息后]

AI: "我将按照 RFC 标准格式编写:
- 标题和作者
- 状态
- 背景
- 目标
- 方案对比
- 建议方案
- 实施计划
- 风险和缓解措施"
```

## 注意事项

1. **不要假设** - 始终先询问关键信息
2. **迭代改进** - 不要期望一次完美
3. **保持简洁** - 清晰胜过华丽
4. **关注读者** - 始终考虑受众需求
5. **验证理解** - 定期确认是否理解正确
